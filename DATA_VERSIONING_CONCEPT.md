# Концепция хранения и версионирования данных

## Общие принципы

Проект bank-products-static использует простую, но эффективную систему хранения и версионирования данных, основанную на следующих принципах:

1. **Прозрачность** - вся информация хранится в открытом формате
2. **Простота** - минимальная сложность архитектуры
3. **Надежность** - использование Git для контроля версий
4. **Масштабируемость** - возможность легко добавлять новые продукты и банки
5. **Историчность** - возможность отследить изменения во времени

## Структура хранения данных

### 1. Основная структура директорий

```
src/
└── data/
    ├── pdfs/                           # Основные банковские продукты
    │   ├── credit-cards/              # Кредитные карты
    │   │   ├── [bank-name]/          # Продукты конкретного банка
    │   │   │   ├── [product-name]-[YYYY-MM-DD].json
    │   │   │   └── ...
    │   │   └── ...
    │   ├── debit-cards/               # Дебетовые карты
    │   │   └── [bank-name]/          # Продукты конкретного банка
    │   │       ├── [product-name]-[YYYY-MM-DD].json
    │   │       └── ...
    │   ├── credits/                   # Кредиты
    │   │   └── [bank-name]/          # Продукты конкретного банка
    │   │       ├── [product-name]-[YYYY-MM-DD].json
    │   │       └── ...
    │   ├── deposits/                  # Вклады
    │   │   └── [bank-name]/          # Продукты конкретного банка
    │   │       ├── [product-name]-[YYYY-MM-DD].json
    │   │       └── ...
    │   └── [bank-name]/              # Продукты конкретного банка (общая директория)
    │       ├── [product-name]-[YYYY-MM-DD].json
    │       └── ...
    ├── banks.json                     # Информация о банках
    ├── categories.json                # Категории продуктов
    └── archive/                       # Архивные продукты
        ├── [year]/                   # Год архивации
        │   └── [bank-name]/          # Банк
        │       └── [product-name]-[YYYY-MM-DD].json
        └── ...
```

### 2. Структура JSON файла продукта

Каждый банковский продукт представлен в виде JSON файла со следующей структурой:

```json
{
  "id": "уникальный-идентификатор-продукта",
  "type": "тип-продукта",
  "bankName": "название-банка",
  "title": "название-продукта",
  "version": {
    "date": "YYYY-MM-DD",
    "source": "источник-информации",
    "updatedBy": "автор-изменений"
  },
  "parameters": {
    "main": {
      "ключевой-параметр": "значение"
    },
    "fees": {
      "комиссия": "значение"
    },
    "requirements": {
      "требование": "значение"
    }
  },
  "validFrom": "YYYY-MM-DD",
  "validUntil": "YYYY-MM-DD (если продукт прекращен)",
  "status": "active|discontinued|updated"
}
```

## Система версионирования

### 1. Именование файлов

Файлы именуются по следующему шаблону:
```
[bank-name]-[product-type]-[product-name]-[YYYY-MM-DD].json
```

Примеры:
- `sberbank-credit-cards-platinum-2023-12-01.json`
- `tinkoff-debit-cards-young-2024-01-15.json`
- `vtb-credits-auto-loan-2024-02-20.json`

### 2. Управление версиями через Git

- Каждое изменение продукта фиксируется в Git
- Используется семантическое версионирование коммитов
- Для отслеживания изменений используется `git log --follow`

### 3. Архивация устаревших версий

- При обновлении продукта старая версия перемещается в архив
- Архивные версии хранятся в `src/data/archive/[year]/`
- Архивные файлы сохраняют оригинальное имя с датой

## Жизненный цикл продукта

### 1. Добавление нового продукта

1. Создается JSON файл с информацией о продукте
2. Файл размещается в соответствующей поддиректории
3. Устанавливается статус `active`
4. Указывается дата начала действия `validFrom`
5. Указывается дата обновления `version.date`

### 2. Обновление продукта

1. Создается новая версия файла с новой датой в имени
2. В старой версии устанавливается `validUntil` и статус `updated`
3. В новой версии устанавливается статус `active`
4. Обе версии фиксируются в Git

### 3. Прекращение продукта

1. В активной версии устанавливается `validUntil` и статус `discontinued`
2. При необходимости создается архивная копия
3. Продукт перемещается в архивную директорию

## Проверка целостности данных

### 1. Валидация структуры

- Проверка соответствия JSON схеме
- Проверка обязательных полей
- Проверка формата значений

### 2. Проверка дубликатов

- Проверка на наличие продукта с тем же ID
- Проверка на схожесть параметров с существующими продуктами
- Использование LLM для проверки дубликатов

### 3. Проверка актуальности

- Периодическая проверка через LLM
- Сравнение с официальными источниками
- Автоматическое определение устаревших данных

## Автоматизация управления данными

### 1. Скрипты валидации

- `scripts/validate-data.js` - проверка структуры всех JSON файлов
- `scripts/check-duplicates.js` - проверка на дубликаты
- `scripts/verify-actual.js` - проверка актуальности данных

### 2. Скрипты версионирования

- `scripts/version-manager.js` - автоматическое управление версиями
- `scripts/archive-manager.js` - автоматическая архивация устаревших версий
- `scripts/update-notifier.js` - уведомление об устаревших данных

## Безопасность и доступ

### 1. Контроль доступа

- Только авторизованные участники могут изменять данные
- Все изменения проходят через pull request
- Обязательное ревью перед слиянием

### 2. Резервное копирование

- Все данные хранятся в Git репозитории
- Используется GitHub/GitLab для внешнего резервного копирования
- Регулярные бэкапы локальных репозиториев

## Миграция данных

### 1. Обновление структуры

- При изменении структуры данных создаются скрипты миграции
- Скрипты автоматически преобразуют старые форматы
- Проверка корректности миграции перед применением

### 2. Совместимость

- Поддержка нескольких версий формата одновременно
- Постепенная миграция без остановки сервиса
- Обратная совместимость при разумных пределах

## Мониторинг и аудит

### 1. Логирование изменений

- Все изменения фиксируются в Git
- Дополнительно ведется лог в `logs/data-changes.log`
- Метрики качества данных в `metrics/data-quality.json`

### 2. Отчетность

- Еженедельные отчеты о состоянии данных
- Отчеты об устаревших продуктах
- Отчеты о качестве и достоверности информации

## Рекомендации по использованию

1. **Последовательность именования** - строго следуйте шаблону именования файлов
2. **Документирование изменений** - всегда указывайте источник и причину изменений
3. **Регулярная проверка** - периодически проверяйте актуальность данных
4. **Архивация** - не удаляйте старые версии, а архивируйте их
5. **Автоматизация** - используйте скрипты для рутинных операций
